## 1. Общая логика работы

В проекте велась параллельная разработка двух реализаций одной идеи:
- `gpt/` - DeepSeek/GPT-поток;
- `qwen/` - Qwen-поток.

Обе версии решают одну задачу:
- парсят `timing.batyrshin.name`;
- сохраняют данные в PostgreSQL;
- поднимают REST API на Django REST Framework;
- дают CLI для работы через API.

## 6. Какой инструмент лучше и для каких задач

- `DeepSeek ` лучше показал себя в задачах, где нужен линейный инженерный каркас: сначала модели, потом API, потом CLI, потом тесты.
- `Qwen` оказался сильнее там, где нужно было быстро предложить альтернативную структуру проекта и более "продуктовое" расширение функциональности, например `results` и AI-анализ заезда.
- Для строгого следования ТЗ и поэтапной реализации удобнее использовать `DeepSeek `.
- Для генерации альтернативных идей, отличающейся CLI-архитектуры и дополнительных возможностей удобнее использовать `Qwen`.

## 7. Советы по промпт-инжинирингу

- На дебаге лучше всего работали промпты с конкретным симптомом: что сломано, где именно и какого поведения ожидаем.
- Если нужна отличающаяся вторая реализация, это нужно явно прописывать: "не копируй первую структуру", "предложи альтернативную архитектуру", "разведи набор команд и API".
- При работе с парсером полезно давать модели конкретные URL страниц и просить описать стратегию извлечения данных отдельно для каждого типа страниц.

## 8. Рекомендации для будущих проектов

- Сохранять все диалоги с ИИ сразу в отдельный markdown-файл, а не восстанавливать их по коммитам постфактум.
- Фиксировать после каждой крупной итерации: какой был промпт, что ответила модель, что реально приняли в код.
- С самого начала разводить две реализации по разным папкам и по разным наборам фич, чтобы сравнение было содержательнее.
- Раньше подключать тесты, потому что именно тесты быстрее всего показывают слабые места сгенерированного API и CLI.
- Для задач с парсингом сразу сохранять реальные примеры HTML или снимки страниц, чтобы потом проще дебажить нестабильную разметку.
